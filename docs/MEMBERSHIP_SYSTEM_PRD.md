## 二、业务目标

小程序主体功能聚焦四大模块：

1. **会员管理** — 199/299 两档付费会员，开卡赠送积分
2. **商品展示** — 在售商品展示（已有 shunwei-api 展示型商品模块）
3. **积分商城** — 积分兑换手机壳/贴膜/数码配件，到店自提核销
4. **现金券** — 储值型消费券，本店+异业商家均可核销

---

<span style="color:red">不涉及快递发货</span>**，积分兑换与现金券均为线下核销模式。

## 三、核心规则（已确认）

### 3.1 会员档位

| 档位 | 价格 | 开卡赠送积分 | 有效期 |
|------|------|-------------|--------|
| 199 会员 | 199 元 | 199,000 积分 | 1 年 |
| 299 会员 | 299 元 | 299,000 积分 | 1 年 |

- **积分汇率**：1 元 = 1,000 积分
- **199 vs 299 差异**（MVP1）：仅等级标识 + 赠送积分数量，折扣/专属商品后续迭代
- **获取通道（双通道）**：
  - 线下：店员录入消费 → 三级审批 → 自动开通
  - 自购：用户微信支付购买 CRMEB 付费会员 → 回调自动赠积分

### 3.2 积分规则

| 积分类型 | 有效期 | 说明 |
|---------|--------|------|
| 赠送积分（开卡赠） | 1 年 | 各批次独立计时 |
| 充值积分（微信购买） | **永久有效** | 1 元 = 1,000 积分 |
| 消耗顺序 | FIFO | 先到期先扣 |

- 会员到期后：会员标识降级；赠送积分按批次过期扣减；现金券 1 年到期冻结

### 3.3 线下消费档位（左闭右开）

店员录入客户线下消费金额，系统自动匹配权益包：

| 消费金额 | 赠送会员 | 现金券 | 赠送积分 |
|---------|---------|--------|---------|
| [2,000 , 3,000) | 199 会员 | 100 元 | 199,000 |
| [3,000 , 6,000) | 199 会员 | 300 元 | 199,000 |
| [6,000 , 10,000) | 299 会员 | 500 元 | 299,000 |
| [10,000 , +∞) | 299 会员 | 800 元 | 299,000 |

**示例**：
- 消费 3,550 元 → 第二档 → 199 会员 + 300 元券 + 199,000 积分
- 消费 5,500 元 → 第二档 → 199 会员 + 300 元券 + 199,000 积分
- 消费 6,500 元 → 第三档 → 299 会员 + 500 元券 + 299,000 积分

### 3.4 权益累加规则

| 维度 | 规则 |
|------|------|
| 单笔消费 | 只匹配一档 |
| 跨笔消费 | 会员、现金券、赠送积分 **均可累加** |
| 会员等级 | 取最高档（199→299 升级），有效期从最近开通日起顺延 1 年 |
| 赠送积分 | 各次积分余额累加，每批次独立 1 年有效 |
| 现金券 | 各次发放金额累加至钱包余额 |
| 防刷单 | 同一小票/单据号不可重复提交 |

**跨笔累加示例**：
1. 先消费 3,550 → 199 会员 + 300 元券 + 199,000 积分
2. 再消费 5,500 → 再得 199 会员 + 300 元券 + 199,000 积分（全部累加）

## 四、角色与权限

### 4.1 角色定义

| 角色 | 来源 | 职责 |
|------|------|------|
| 超级管理员 | CRMEB 管理员 | 终审批、开通异业商家、配置规则、上架积分商品、开关免审 |
| 店长 | 按门店 `division_id`，每店 1-2 人，超管指定 | 二级审批 |
| 店员 | 原 CRMEB 分销员改造（`is_staff`） | 录入消费、发起审批、核销积分兑换 |
| 异业商家 | 超管手动开通 | 核销现金券、查看待结算金额 |
| 普通用户 | 小程序注册用户 | 查看会员/积分/现金券、兑换、自购会员 |

### 4.2 三级审批流（MVP2）

```
店员录入消费金额 + 凭证
    ↓
系统自动匹配档位权益包
    ↓
店长审批（通过/驳回+原因）
    ↓
超管终批
    ↓
自动开通会员 + 发放积分 + 发放现金券
```

- **驳回后重提**：店员可修改后重新提交，展示驳回原因
- **无小票场景**：不填小票号，走人工审核（店长+超管重点审核）
- **消息通知**：MVP 用站内待办/红点；微信订阅消息后期实现

### 4.3 积分商城核销

- **仅超管可上架/下架**积分商品
- 前端 **不展示库存数量**；库存为 0 时提示「暂时无法兑换，过两天试试」
- 到店自提，不发快递
- **免审开关**：超管可打开/关闭；关闭时走三级审批（店员→店长→超管）

### 4.4 现金券规则

- **储值型钱包**：可部分核销（如 100 元券核销 66 元，剩 34 元）
- **核销模式**：超管可切换「任意金额」或「整百核销」
- **使用范围**：异业商家 + 本店均可核销，全平台通用
- **有效期**：1 年，到期冻结
- **商家结算**：仅展示待结算金额 + 已结算状态；**无小程序内提现**，线下打款

### 4.5 异业商家

- **入驻方式**：超管手动开通账号与核销权限
- **适用场景**：火锅店、服装店、电影院、超市等合作实体店
- **结算**：核销金额全额计入商家待结算台账，T+3 线下结算

## 五、技术架构

### 5.1 系统分工

```
微信小程序
  ├─ 会员/积分/积分商城 ──→ CRMEB（复用 eb_user、积分商城）
  ├─ 现金券/审批/异业商家 ──→ shunwei-api（新建模块）
  └─ 商品展示 ──→ shunwei-api（已有展示型商品）
```

### 5.2 CRMEB 可复用 vs 需新建

| 需求 | CRMEB 现状 | 处理方式 |
|------|-----------|---------|
| 199/299 付费会员 | `eb_member_ship` 支持多卡型 | ✅ 配置 + 开卡赠积分钩子 |
| 赠送积分 | `eb_user.integral` + 积分流水 | ✅ 直接发放 |
| 积分商城兑换 | `points_mall` + 自提核销 | ✅ 改配送为到店自提 |
| 积分充值 | 无积分直充 | ❌ shunwei-api 新建 |
| 积分批次 FIFO | 单字段无批次 | ❌ shunwei-api `integral_batch` 表 |
| 分销员改店员 | `is_staff` + 事业部 | ⚠️ UI/权限改造 |
| 储值型现金券 | 优惠券一次性核销 | ❌ shunwei-api 现金券钱包 |
| 三级审批 | 无审批链 | ❌ shunwei-api 新建 |
| 异业商家 | 无此体系 | ❌ shunwei-api 新建 |

## 十、方案优化建议（v4.2 审查）

> 以下不推翻已确认规则，仅补充边界场景与可落地优化，标注 **需拍板** 的项。

### 10.1 业务规则补全（✅ 用户已确认按建议执行）

| # | 场景 | 现状 | 建议 |
|---|------|------|------|
| 1 | 消费不足 2000 元 | 未定义 | 不匹配任何档位，页面提示「未达权益门槛」 |
| 2 | 边界金额（如 3000、6000、10000） | 左闭右开 | 3000→第二档、6000→第三档；**录入页实时预览**匹配结果 |
| 3 | 已有 299 会员再消费 3500 | 跨笔累加 | **取高不降级**：保持 299，只发 300 元券+199000 积分，不重复开 199 |
| 4 | 用户微信自购 199 + 线下再赠 199 | 双通道 | 会员有效期取 `max(当前到期, 新开通+1年)`，积分/券照常累加 |
| 5 | 终批误操作 | 未定义 | 超管 **24 小时内可撤销**，回滚会员/积分/券并记审计日志 |
| 6 | 现金券累加 | 钱包余额 | 建议 **按批次记录**（每笔发放独立 1 年到期），核销 FIFO，避免「永远用不完旧券」 |
| 7 | 积分商城定价 | 未写 | 手机壳类建议 **5000–20000 积分**（5–20 元等值），超管后台配置 |
| 8 | 店员看客户权益 | 未定义 | 员工工作台展示：会员等级、到期日、积分余额、现金券余额（只读） |

### 10.2 分期微调（更快上线）

当前 MVP1 范围合理。建议 **MVP1 先不做「免审关闭时的三级审批」**，该能力随 MVP2 审批流一起做，避免 MVP1 做半套审批：

| 调整 | 说明 |
|------|------|
| MVP1 免审开关 | 默认 **开启**（店员直接核销积分兑换） |
| MVP1 不做 | 关闭免审后的三级审批（挪到 MVP2 与消费审批统一） |
| MVP2 合并 | 消费审批 + 积分兑换审批共用同一套审批引擎 |

### 10.3 技术风险与对策

| 风险 | 对策 |
|------|------|
| `integral_batch` 与 `eb_user.integral` 不一致 | 所有积分变动走 shunwei-api 统一入口，事务内双写 + 日对账任务 |
| 小程序是构建产物 | MVP1 优先改 **shunwei-api 后台 + CRMEB 后台**；小程序只改必要页面 |
| 生产 JWT 校验 | 必须配置生产 `CRMEB_APP_KEY`，否则本地测通、上线登录失败 |
| 储值型现金券合规 | 建议法务确认预付卡/单用途卡备案要求（**不阻塞 MVP1**） |

### 10.4 可选优化（不阻塞，后续迭代）

- 审批超时提醒（48h 未批自动提醒店长/超管）
- 异业商家核销后用户收到站内消息
- 积分/现金券即将过期前 7 天提醒
- 后台导出：审批记录、核销记录、商家结算台账

## 十一、本地测试与上线部署策略 ⚠️

### 11.1 本地测试（你现在做的）

```
1. 导入 SQL → 本机 MySQL（so1988_shunwei）
2. 配置 shunwei-api .env（库名/账号/APP_KEY）
3. 启动 shunwei-api + CRMEB（可选）
4. 微信开发者工具导入小程序，勾选「不校验合法域名」
5. 用测试账号验证：会员开通 → 积分发放 → 积分兑换
```

### 11.2 上线时「上传覆盖」—— 正确 vs 错误

| 操作 | ✅ 正确 | ❌ 错误（会丢数据） |
|------|--------|-------------------|
| **代码** | 上传 shunwei-api 新代码、小程序新版本 | — |
| **数据库** | 只在生产库 **执行增量 DDL**（`CREATE TABLE` 新表、`ALTER` 新字段） | 用本地测试库 **整库覆盖** 生产库 |
| **CRMEB 配置** | 后台手动添加 199/299 会员档位、积分商品 | 覆盖 `eb_system_config` 等配置表 |
| **备份** | 上线前 ** mysqldump 备份生产库** | 无备份直接操作 |

**关键原则**：
- 本地 SQL 文件 = **开发参考 + 联调**，不是上线包
- 上线只带：**代码 + 迁移脚本**（架构师 DDL），不带整库
- 生产库已有真实订单/用户/支付数据，**绝不能被测试库覆盖**

### 11.3 推荐上线步骤

```
1. 生产库全量备份（mysqldump）
2. 生产执行 MVP1 迁移脚本（仅 shunwei-api 新表 + 必要 ALTER）
3. CRMEB 后台配置 199/299 会员档、积分商品
4. 部署 shunwei-api（PM2 + Nginx 反代）
5. 上传小程序新版本 → 微信提审/发布
6.  smoke test：1 个测试账号走通开卡赠积分 → 积分兑换
7. 确认无误后开放给店员使用
```

### 11.4 本地与生产环境对照

| 项 | 本地 | 生产 |
|----|------|------|
| DB | 本机 MySQL + 脱敏/快照 | 服务器 MySQL（**只跑增量脚本**） |
| shunwei-api | 127.0.0.1:8787 | Nginx 反代 HTTPS 域名 |
| 小程序 API | 不校验合法域名 | 需配置 request 合法域名 |
| APP_KEY | 必须与生产一致 | crmeb/.env 中的 APP_KEY |
