# 顺为小程序「重构小程序端 + 后台端」总体规划（先规划后实施）

> 决策背景：CRMEB 版本不匹配（生产 v5.6.4 vs 重编译 v6.0.1），与其纠缠升级，
> 决定**重构小程序端 + 后台端**：以现有后台数据库为基础迁移真实数据，
> **保留微信支付/登录等微信相关接口**，其余按现有业务需求重做。
> 本文档为**规划**，经你评审确认后再进入实施。

---

## 一、为什么重构是可行且划算的（关键调研结论）

对现有生产库 `so1988_shunwei` 实测：

| 数据 | 实际量 | 含义 |
|------|-------:|------|
| `eb_user`（用户）| **1770** | 真实用户资产，**必须迁移** |
| `eb_wechat_user`（微信绑定）| **1768** | 几乎全是微信用户，**登录强依赖微信** |
| `eb_user_bill`（积分/余额流水）| **600** | 积分历史，需迁移 |
| `eb_store_order`（商城订单）| **0** | ⭐ **没有线上订单**，无快递业务 |
| `eb_store_product`（商品）| 12 | 展示型商品，少量 |
| `eb_store_integral`（积分商品）| 4 | 积分商城商品，少量 |
| `eb_member_ship`（会员卡）| 7 | 含我们新增的 SW199/SW299 |
| `eb_system_store`（门店）| 0 | 门店待录入 |

**结论**：
1. 真正有价值、必须保的资产 = **1770 用户 + 1768 微信绑定 + 600 积分流水 + 已建的会员/积分数据**。
2. **没有订单/快递业务**（store_order=0），与你"全部线下核销、不发快递"的模式一致 → 重构可**砍掉 CRMEB 绝大部分电商/物流/分销复杂度**。
3. 你的核心新功能（会员/积分/现金券/异业商家）**已在 `shunwei-api` 实现**，可直接作为新系统的后端底座。

→ 所以重构不是从零，而是**「精简重做小程序壳 + 后台，复用 shunwei-api + 现有数据库」**，工作量与风险都可控。

### 微信接口「保留」的两种实现方案（待你选）
- **方案甲（推荐，省事）**：保留 CRMEB 的 `crmeb/` 仅作为「微信网关」——只用它的 `v2/wechat/routine_auth`(登录)、`order/pay`(支付下单)、支付回调；其余 CRMEB 功能不再使用。小程序登录/支付走 CRMEB，业务走 shunwei-api。
- **方案乙（更彻底）**：在 shunwei-api 内新建轻量 `wechat` 模块，直接对接微信官方 `code2session` + 微信支付 v3 API，完全弃用 CRMEB。需重配微信支付证书。

> 51 个微信相关配置（AppID/AppSecret/mch_id/支付密钥）在 `eb_system_config`，两方案都基于这些凭证。

## 四、数据迁移方案

**原则**：现有库为**数据源**，重构系统**沿用同一套 `eb_user` 等核心表**（不改表名、不重导），只在其上叠加 `sw_*`。这样：
- 1770 用户、1768 微信绑定、积分流水**零迁移成本**（本来就在用）。
- 微信 openid/unionid 关系不动 → 老用户**无感登录**。

需要做的"迁移/清洗"：
1. **会员卡**：确认 SW199/SW299 配置（已建 id6/7 或 id10/11，需统一）。
2. **积分对齐**：把 `eb_user.integral` 历史积分纳入 `sw_integral_batch`（给老用户建一个"历史积分"批次，便于 FIFO 统一管理）。
3. **角色映射**：把原"分销员/店员"(`is_staff`/`spread`)映射为新"店面员工"角色（MVP2 已规划）。
4. **冗余清理**：废弃的电商/物流/分销表标记为不使用（不删，留存历史）。

## 九、风险与注意

1. **微信登录连续性**：必须保证 1768 个老用户 openid 不变即可无感登录 → 沿用同库 `eb_wechat_user` 是关键，不可重置。
2. **微信支付证书**：方案乙需重新配置 mch 证书；方案甲沿用 CRMEB 现有配置最稳。
3. **数据不重导**：直接在生产库上叠加 `sw_*`，老数据零迁移；上线前务必**备份**。
4. **并行不影响线上**：重构期间老小程序(v5.6.4)继续在线，新系统在测试域名/测试小程序验证，切换时只改小程序发布版本 + 后端路由。
5. **审批/核销是新核心**：CRMEB 无此能力，全部新建，是工作量大头（P3/P4）。
